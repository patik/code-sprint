Startup Mentality

  • Lean team
  • Focused on specific skills
  • One simple goal
Instead, we can look to startups for inspiration. They get a lot done with fewer resources. Some of them even disrupt entire industries. With ITS we There are a lot of good developers out there -- if you let them focus on one specific project they can flourish.

Recipe for a Code Sprint

  • Small, dedicated team
  • Single goal
  • Short time frame
  • No distractions
Putting together a SWAT team is much easier now with ITS since we can draw resources from across clusters. Give that team a clear, singular goal, and take away their distractions. Make it a good goal, something that will produce real, usable results. You can put other projects aside for a week, it won't kill you.

Why Now?

  • Consolidation
  • Connectedness
So why am I talking about this now in 2013? Why hasn't this been the standard for years? To put it simply, we have GitHub. More specifically, developers tools have matured and are no largely cross-platform. A lot of things are web-based. We have one place with all of our code. One place for issue tracking. We are not tied to our desks. We can work in the same room, or remotely. Each has its benefits, but ultimately everything comes back to one place.

A 3-week code sprint focused on responsive web design. Not only did we make a proof of concept site, Go Responsive, to show how several different UI patterns can work in a responsive manner... We even redesigned an entire agency site, the Division of Tax Appeals. And if that wasn't enough, we put together an entire toolkit for making responsive sites. That's already been used to make several more NYS sites.

In the Trenches

My experience working on the responsive web design SWAT team was incredible. It was pretty much non-stop action for the whole three weeks. About a dozen of us got together for it. We started off brainstorming ideas about what designing responsively means, what a web site needs to perform well across a multitude of devices. We organized all those things on a big poster board and started to code.

In the Trenches

We knew we were expected to make a proof-of-concept site, and, if we could, a framework for making more sites. That clear focus kept us on task. There are so many rabbit holes that are very tempting for developers to jump down -- performance optimizations, adding extra bells and whistles. (Not testing, though, we let the users do that.) But with a well-defined goal, and a project lead to keep us on task, we were able to stay on target.

In the Trenches

As I said, this was a pretty intense 3 weeks. That intensity drove all of us to keep working. The deadline loomed over us, but in a good way, a way that drove us to keep working -- we all *wanted* this project to succeed. We knew that responsive design was the way the state needed to go, and this was our chance to prove it would work. That desire was instrumental to our success.

Keys to Success

  1. Well-defined goal
  2. Skilled team members
  3. Time limit
Really, the whole thing boils down to these three points. Managers, if you can provide this, your team will thrive. Your good developers will do great, and your great developers will be off the charts.

Where it Works Best

  • Exploration
  • Low-hanging fruit
  • Projects that never seem to end
Code sprints are a great opportunity to explore an area or a direction you've been thinking about getting into. Invest a few weeks to flesh it out and see if that direction works for you. It's also great for picking off those items you know you need but just haven't gotten around to. Need to switch from Subversion to Git? Need to add a JSON option to your API? Set up a code sprint and just crank through it.

Putting Together a Code Sprint

  • 2-3 week project
  • Round up developers
  • Put other projects into maintenance mode
To put together a code sprint of your own, find a project that could be done in a couple weeks if only everyone could devote their time to it. Find people with the right skills. Be sure to get the right variety for the project. And make sure these people can plan to spend the whole time, not just a half a day, working on the sprint.

Preparation

  • Workplace logistics (parking, security passes, network access)
  • Procure laptops and software, or BYOD
  • Setup the development stack
Before beginning the sprint, make sure you've worked out all the little things that are tangential to actual work. Let the developers get right down to business from the first minute. If you know what software you'll need ahead of time, getting that all sorted out so you don't waste time creating GitHub accounts or configuring IDEs.
